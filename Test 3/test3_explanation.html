1.2 Why This Three-Layer Architecture?

Each layer has a single, clearly bounded responsibility. This separation yields four concrete benefits: debuggability (isolate failures to one layer), reusability (multiple mart models can share one intermediate model), testability (data quality checks can be scoped per layer), and maintainability (changing business logic in one model does not break unrelated models).

2   Staging Layer (stg_*)

Clean mirror of source — rename, cast, null-handle. Zero business logic.

2.1 Design Rules Applied to All Staging Models

RuleImplementationRationale
Rename columns Source names → clear snake_case
(e.g., order_purchase_timestamp → purchased_at)		 Downstream models reference clean names; source typos are fixed once here
Explicit type casts price::numeric(10,2), review_score::int Prevents implicit cast errors in downstream aggregations; makes schema self-documenting
Null filtering WHERE order_id IS NOT NULL Orphaned records without PKs would corrupt downstream joins
No business logic No CASE WHEN, no joins between source tables (except stg_products) Keeps staging as a stable, predictable foundation; business logic changes belong in int_*
Empty string → NULL NULLIF(TRIM(review_comment_title), '') Standardises absence of data; avoids '' vs NULL inconsistency in analytics

2.2 Notable Design Decisions Per Model

stg_products — Exception: One JOIN Allowed

Why stg_products joins the translation table

The product_category_name_translation table is a pure reference/lookup table with no business semantics — it simply maps Portuguese names to English equivalents. Including this join in staging eliminates the need to replicate it in every downstream model that needs English category names. A LEFT JOIN is used (not INNER) to preserve products with untranslated categories, which fall back to 'uncategorized' via COALESCE.

Exception criteria: A staging JOIN is acceptable when the joined table is a static reference/lookup (no business logic, rarely changes, no PK ambiguity).

stg_products — Typo Fix in Source Column Names

-- Source has typos: "lenght" instead of "length" (missing 't')
p.product_name_lenght        AS product_name_length,
p.product_description_lenght AS description_length

The original Olist dataset has misspelled column names. Staging is the only place where these are corrected — all downstream models use the clean _length spelling.

stg_order_reviews — Derived boolean: is_low_score

(review_score <= 2) AS is_low_score

This is the only derived column in staging. It is placed here because: (1) it is a simple, non-contextual rule (score ≤ 2 = poor), (2) it will be consumed by multiple downstream models (int_orders_enriched, mart_product_performance), and (3) it requires no joins to compute. If the threshold were to change, it would only need updating in one place.

stg_order_payments — Keeps Raw Payment Grain

Design Note: One Row Per Payment Record A single order can have multiple payment rows (e.g., credit card + voucher, or instalments recorded separately). stg_order_payments preserves this grain. Aggregation to order-level (total_payment, primary_payment_type) happens in int_orders_enriched, where the business context of "per-order payment" is appropriate.

    order_status,

3   Intermediate Layer (int_*)

Business logic, cross-model joins, and derived metrics. The "facts" of the warehouse.

3.1 Design Philosophy

The intermediate layer is where domain knowledge lives. Models here answer the question: "What does the business need to know about each entity, combining all relevant context?" Each intermediate model has a clearly defined grain (one row per order, seller, or customer) and provides reusable, pre-computed metrics that multiple mart models can consume.

Intermediate models are never queried directly by BI tools. They are stepping stones — too granular for a Senior Manager dashboard, but too aggregated for a staging model. Their purpose is to enable mart models to be simple SELECT + GROUP BY operations.

3.2 int_orders_enriched — The Central Fact Model

Grain: One row per order_id

Sources joined: stg_orders ⋈ stg_customers ⋈ stg_order_items (aggregated) ⋈ stg_order_payments (aggregated) ⋈ stg_order_reviews (aggregated)

Key Derived Metrics

ColumnFormulaBusiness Meaning
delivery_delay_days EPOCH(delivered_at - estimated_delivery_at) / 86400 Positive = late by N days. Negative = early. NULL = not yet delivered. Rounds to 1 decimal for precision without noise.
is_late_delivery delivered_at > estimated_delivery_at Boolean flag for easy COUNT/SUM in aggregations downstream. Avoids repeating the date comparison in every mart model.
total_delivery_days EPOCH(delivered_at - purchased_at) / 86400 End-to-end fulfilment time from customer perspective. Different from delay_days which measures against the promise.
primary_payment_type Payment type with highest payment_amount in order Identifies dominant payment method for orders split across multiple payment types.
has_low_score BOOL_OR(is_low_score) from review_agg TRUE if any review for this order is 1–2 stars. Used as a quality flag in product and seller analytics.

Why Use CTEs Within the Model (not subqueries)?

WITH order_agg AS (
    SELECT order_id, SUM(item_price) AS total_items_price ...
    FROM stg_order_items GROUP BY order_id
),
payment_agg AS ( ... ),
review_agg  AS ( ... )
SELECT o.*, c.*, oa.*, pa.*, ra.*, -- delivery metrics ...
FROM stg_orders o
JOIN stg_customers c ON ...
LEFT JOIN order_agg oa ON oa.order_id = o.order_id ...

Each upstream model (order_items, payments, reviews) is aggregated to order-grain in its own CTE before joining to orders. This prevents the fan-out problem — if orders were joined to non-aggregated items first, each order would appear N times (once per item), causing inflated aggregations downstream.

3.3 int_seller_metrics — Operational Performance per Seller

Grain: One row per seller_id

Aggregates all completed orders containing items from each seller. Provides: revenue, volume, customer satisfaction (avg review score, low-score rate), and delivery performance (late rate, avg days late). Also identifies each seller's top category by revenue using DISTINCT ON.

3.4 int_customer_metrics — Lifetime Behaviour per Customer

Grain: One row per customer_unique_id

Derives RFM-adjacent metrics: total orders, total spend, first/last order dates, customer lifetime days, avg days between purchases (frequency), and preferred state. These feed directly into mart_customer_segments.

avg_days_between_orders uses (lifetime_days) / (total_orders - 1) to measure true inter-purchase intervals, and returns NULL for one-time buyers (where the denominator would be 0).

3.5 Full SQL — int_orders_enriched.sql

Central fact model — one row per order, consumed by all mart models. Pre-aggregates items, payments, and reviews to order grain before joining to prevent fan-out.

-- Model: int_orders_enriched | Grain: one row per order_id
-- Sources: stg_orders, stg_customers, stg_order_items, stg_order_payments, stg_order_reviews

WITH order_agg AS (
    SELECT
        order_id,
        ROUND(SUM(item_price)::numeric,  2) AS total_items_price,
        ROUND(SUM(freight_cost)::numeric, 2) AS total_freight,
        COUNT(item_sequence)                 AS item_count
    FROM stg_order_items
    GROUP BY order_id
),
payment_agg AS (
    SELECT
        order_id,
        ROUND(SUM(payment_amount)::numeric, 2) AS total_payment,
        MAX(installments)                       AS max_installments,
        (   -- Dominant payment type = highest individual payment amount
            SELECT payment_type
            FROM stg_order_payments sp2
            WHERE sp2.order_id = sp.order_id
            ORDER BY payment_amount DESC
            LIMIT 1
        ) AS primary_payment_type
    FROM stg_order_payments sp
    GROUP BY order_id
),
review_agg AS (
    SELECT
        order_id,
        ROUND(AVG(score)::numeric, 1) AS avg_review_score,
        BOOL_OR(is_low_score)         AS has_low_score,
        COUNT(*)                      AS review_count
    FROM stg_order_reviews
    GROUP BY order_id
)
SELECT
    o.order_id,
    o.customer_id,
    c.customer_unique_id,
    c.city            AS customer_city,
    c.state           AS customer_state,
    o.order_status,
    o.purchased_at,    o.approved_at,    o.shipped_at,
    o.delivered_at,    o.estimated_delivery_at,
    COALESCE(oa.total_items_price, 0) AS total_items_price,
    COALESCE(oa.total_freight,     0) AS total_freight,
    COALESCE(oa.item_count,        0) AS item_count,
    COALESCE(pa.total_payment,     0) AS total_payment,
    pa.primary_payment_type,
    COALESCE(pa.max_installments,  1) AS max_installments,
    ra.avg_review_score,
    COALESCE(ra.has_low_score, FALSE) AS has_low_score,
    COALESCE(ra.review_count,  0)     AS review_count,

    -- Delivery performance (positive = late, negative = early, NULL = not yet delivered)
    CASE
        WHEN o.delivered_at IS NOT NULL AND o.estimated_delivery_at IS NOT NULL
        THEN ROUND(EXTRACT(EPOCH FROM (o.delivered_at - o.estimated_delivery_at)) / 86400.0, 1)
    END AS delivery_delay_days,

    CASE WHEN o.delivered_at > o.estimated_delivery_at THEN TRUE ELSE FALSE
    END AS is_late_delivery,

    CASE
        WHEN o.delivered_at IS NOT NULL AND o.purchased_at IS NOT NULL
        THEN ROUND(EXTRACT(EPOCH FROM (o.delivered_at - o.purchased_at)) / 86400.0, 1)
    END AS total_delivery_days

FROM stg_orders o
JOIN stg_customers c     ON c.customer_id = o.customer_id
LEFT JOIN order_agg oa   ON oa.order_id   = o.order_id
LEFT JOIN payment_agg pa ON pa.order_id   = o.order_id
LEFT JOIN review_agg ra  ON ra.order_id   = o.order_id;

3.6 Full SQL — int_customer_metrics.sql & int_seller_metrics.sql

int_customer_metrics.sql

One row per customer_unique_id. Derives lifetime spend, purchase frequency, and preferred state. Feeds mart_customer_segments.

-- Model: int_customer_metrics | Grain: one row per customer_unique_id
-- Excludes cancelled orders

WITH customer_base AS (
    SELECT customer_unique_id, order_id, purchased_at,
           total_payment, avg_review_score, order_status, customer_state
    FROM int_orders_enriched
    WHERE order_status != 'canceled' AND purchased_at IS NOT NULL
),
customer_state_mode AS (
    -- Most frequent state per customer (DISTINCT ON picks top by window count)
    SELECT DISTINCT ON (customer_unique_id)
        customer_unique_id,
        customer_state AS preferred_state,
        COUNT(*) OVER (PARTITION BY customer_unique_id, customer_state) AS state_orders
    FROM customer_base
    ORDER BY customer_unique_id, state_orders DESC
)
SELECT
    cb.customer_unique_id,
    COUNT(DISTINCT cb.order_id)                    AS total_orders,
    ROUND(SUM(cb.total_payment)::numeric,  2)      AS total_spend,
    ROUND(AVG(cb.total_payment)::numeric,  2)      AS avg_order_value,
    MIN(cb.purchased_at)                           AS first_order_at,
    MAX(cb.purchased_at)                           AS last_order_at,
    ROUND(EXTRACT(DAY FROM (MAX(cb.purchased_at) - MIN(cb.purchased_at)))::numeric, 0)
                                                   AS customer_lifetime_days,
    -- NULL for single-purchase customers (avoid divide-by-zero)
    CASE WHEN COUNT(DISTINCT cb.order_id) > 1
         THEN ROUND(EXTRACT(DAY FROM (MAX(cb.purchased_at) - MIN(cb.purchased_at)))
                    / (COUNT(DISTINCT cb.order_id) - 1.0), 0)
    END                                            AS avg_days_between_orders,
    ROUND(AVG(cb.avg_review_score)::numeric, 2)    AS avg_review_score,
    csm.preferred_state
FROM customer_base cb
LEFT JOIN customer_state_mode csm ON csm.customer_unique_id = cb.customer_unique_id
GROUP BY cb.customer_unique_id, csm.preferred_state
ORDER BY total_spend DESC;

int_seller_metrics.sql

One row per seller_id. Combines revenue, satisfaction, and delivery reliability. Top category identified via DISTINCT ON.

-- Model: int_seller_metrics | Grain: one row per seller_id

WITH seller_base AS (
    SELECT oi.seller_id, oi.order_id, oi.item_price,
           oe.order_status, oe.avg_review_score,
           oe.is_late_delivery, oe.delivery_delay_days, oe.has_low_score
    FROM stg_order_items oi
    JOIN int_orders_enriched oe ON oe.order_id = oi.order_id
    WHERE oe.order_status != 'canceled'
),
seller_top_category AS (
    SELECT DISTINCT ON (oi.seller_id)
        oi.seller_id,
        p.category_name_en AS top_category,
        SUM(oi.item_price) OVER (PARTITION BY oi.seller_id, p.category_name_en) AS category_revenue
    FROM stg_order_items oi
    JOIN stg_products p ON p.product_id = oi.product_id
    JOIN int_orders_enriched oe ON oe.order_id = oi.order_id
    WHERE oe.order_status != 'canceled'
    ORDER BY oi.seller_id, category_revenue DESC
)
SELECT
    sb.seller_id,
    COUNT(DISTINCT sb.order_id)                                   AS total_orders,
    COUNT(*)                                                      AS total_items_sold,
    ROUND(SUM(sb.item_price)::numeric, 2)                         AS total_revenue,
    ROUND(AVG(sb.item_price)::numeric, 2)                         AS avg_item_price,
    ROUND(SUM(sb.item_price)::numeric / NULLIF(COUNT(DISTINCT sb.order_id),0), 2)
                                                                  AS avg_revenue_per_order,
    ROUND(AVG(sb.avg_review_score)::numeric, 2)                   AS avg_review_score,
    COUNT(CASE WHEN sb.has_low_score THEN 1 END)                  AS low_score_orders,
    ROUND(COUNT(CASE WHEN sb.has_low_score THEN 1 END) * 100.0
          / NULLIF(COUNT(DISTINCT sb.order_id),0), 1)             AS low_score_rate_pct,
    COUNT(CASE WHEN sb.is_late_delivery THEN 1 END)               AS late_delivery_count,
    ROUND(COUNT(CASE WHEN sb.is_late_delivery THEN 1 END) * 100.0
          / NULLIF(COUNT(DISTINCT sb.order_id),0), 1)             AS late_delivery_rate_pct,
    ROUND(AVG(CASE WHEN sb.is_late_delivery THEN sb.delivery_delay_days END)::numeric, 1)
                                                                  AS avg_days_late,
    stc.top_category
FROM seller_base sb
LEFT JOIN seller_top_category stc ON stc.seller_id = sb.seller_id
GROUP BY sb.seller_id, stc.top_category
ORDER BY total_revenue DESC;

4   Mart Layer (mart_*)

Business-ready aggregations — directly consumed by BI dashboards and Senior Manager reports

4.1 Design Philosophy

Mart models are designed for BI consumption: wide, flat, pre-aggregated tables that a Senior Manager or a BI tool can query without complex joins. Each mart model answers a specific business question, and column names are chosen for clarity over technical precision.

4.3 mart_customer_segments

Business Question: "Who are our customers and how do we retain them?"

Classifies every customer into one of four segments based on purchase recency and frequency. Designed to drive CRM targeting and retention campaign prioritisation.

SegmentRuleCount (dataset)Avg Spend (R$)
newExactly 1 order ever92,635161.56
returning2+ orders, last order ≤ 180 days ago1,239321.31

Critical Business Insight from Segmentation 96.8% of customers (92,635 out of ~95,500) made only one purchase — they are classified as "new" indefinitely. Returning customers spend 2× more (R$ 321 vs R$ 162) than one-time buyers. This makes retention the highest-ROI business initiative: converting even 1% of new customers to returning adds ~R$ 15M in incremental revenue.

Why use dataset max date (not NOW()) as reference?
The segmentation uses MAX(purchased_at) from the dataset as the reference date instead of CURRENT_TIMESTAMP. This makes the model reproducible: running the same query twice at different times produces the same result for historical analysis. Using NOW() would continuously re-classify customers as time passes, making historical snapshots inconsistent.

4.4 mart_product_performance

Business Question: "Which products drive revenue and which need attention?"

Provides product-level metrics combining commercial performance (revenue, units sold) with quality signals (review scores, low-score rates, late delivery rates). Includes three ranking dimensions:

  • revenue_rank — global rank across all products
  • category_revenue_rank — rank within the product's category (enables "best in category" analysis)
  • quality_adjusted_rank — ranks by avg_review_score × total_revenue, penalising high-revenue products with poor ratings

The quality-adjusted ranking is designed to surface products that are commercially important but risk eroding brand trust — a signal for Senior Managers to investigate product quality issues.

4.5 Full SQL — Mart Models

Complete SQL for all three mart models. Materialized as tables with full refresh strategy — optimized for BI consumption.

mart_monthly_revenue.sql

-- =============================================================================
-- Model  : mart_monthly_revenue  |  Layer: Mart
-- Sources: int_orders_enriched
-- Grain  : one row per calendar month
-- Refresh: full refresh (aggregations are non-idempotent with incremental)
-- =============================================================================

WITH monthly_base AS (
    SELECT
        DATE_TRUNC('month', purchased_at)::date          AS month,
        COUNT(DISTINCT order_id)                          AS total_orders,
        COUNT(DISTINCT customer_unique_id)                AS unique_customers,
        ROUND(SUM(total_payment)::numeric,   2)           AS total_revenue,
        ROUND(AVG(total_payment)::numeric,   2)           AS avg_order_value,
        ROUND(SUM(total_items_price)::numeric, 2)         AS total_items_revenue,
        ROUND(SUM(total_freight)::numeric,   2)           AS total_freight_revenue,
        COUNT(CASE WHEN is_late_delivery THEN 1 END)      AS late_delivery_count,
        ROUND(
            COUNT(CASE WHEN is_late_delivery THEN 1 END) * 100.0
            / NULLIF(COUNT(DISTINCT order_id), 0)
        , 1)                                              AS late_delivery_rate_pct,
        ROUND(AVG(avg_review_score)::numeric, 2)          AS avg_review_score
    FROM int_orders_enriched
    WHERE order_status NOT IN ('canceled', 'unavailable')
      AND purchased_at IS NOT NULL
    GROUP BY 1
)
SELECT
    month, total_orders, unique_customers, total_revenue, avg_order_value,
    total_items_revenue, total_freight_revenue,
    late_delivery_count, late_delivery_rate_pct, avg_review_score,

    -- Month-over-month comparisons
    LAG(total_revenue) OVER (ORDER BY month)             AS prev_month_revenue,
    LAG(total_orders)  OVER (ORDER BY month)             AS prev_month_orders,
    ROUND(
        (total_revenue - LAG(total_revenue) OVER (ORDER BY month)) * 100.0
        / NULLIF(LAG(total_revenue) OVER (ORDER BY month), 0)
    , 2)                                                 AS mom_revenue_growth_pct,
    ROUND(
        (total_orders - LAG(total_orders) OVER (ORDER BY month)) * 100.0
        / NULLIF(LAG(total_orders) OVER (ORDER BY month), 0)
    , 2)                                                 AS mom_orders_growth_pct,

    -- Running totals (year-to-date)
    SUM(total_revenue) OVER (
        PARTITION BY EXTRACT(YEAR FROM month)
        ORDER BY month ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    )                                                    AS ytd_revenue,
    SUM(total_orders) OVER (
        PARTITION BY EXTRACT(YEAR FROM month)
        ORDER BY month ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    )                                                    AS ytd_orders

FROM monthly_base
ORDER BY month;

mart_customer_segments.sql

-- =============================================================================
-- Model  : mart_customer_segments  |  Layer: Mart
-- Sources: int_customer_metrics, int_orders_enriched
-- Grain  : one row per customer_unique_id
-- Note   : Uses MAX(purchased_at) as reference date — reproducible, not NOW()
-- Segments: new (1 order) | returning (<=180d) | at_risk (181-365d) | churned (>365d)
-- =============================================================================

WITH latest_date AS (
    SELECT MAX(purchased_at) AS reference_date
    FROM int_orders_enriched WHERE order_status != 'canceled'
),
customer_data AS (
    SELECT cm.*, ld.reference_date,
        ROUND(EXTRACT(DAY FROM (ld.reference_date - cm.last_order_at))::numeric, 0)
            AS days_since_last_order
    FROM int_customer_metrics cm CROSS JOIN latest_date ld
),
segmented AS (
    SELECT *,
        CASE
            WHEN total_orders = 1                                              THEN 'new'
            WHEN total_orders > 1 AND days_since_last_order <= 180       THEN 'returning'
            WHEN total_orders > 1 AND days_since_last_order BETWEEN 181 AND 365 THEN 'at_risk'
            WHEN total_orders > 1 AND days_since_last_order > 365        THEN 'churned'
            ELSE 'unknown'
        END AS customer_segment,
        NTILE(4) OVER (ORDER BY total_spend DESC) AS spend_quartile
    FROM customer_data
)
SELECT
    customer_unique_id, customer_segment, total_orders,
    ROUND(total_spend::numeric, 2)      AS total_spend,
    ROUND(avg_order_value::numeric, 2)  AS avg_order_value,
    first_order_at, last_order_at, days_since_last_order,
    customer_lifetime_days, avg_days_between_orders,
    ROUND(avg_review_score::numeric, 2) AS avg_review_score,
    preferred_state, spend_quartile,
    CASE spend_quartile
        WHEN 1 THEN 'high_value'    WHEN 2 THEN 'mid_value'
        WHEN 3 THEN 'low_value'     WHEN 4 THEN 'at_risk_value'
    END AS value_tier,
    reference_date AS reference_date_used
FROM segmented
ORDER BY total_spend DESC;

mart_product_performance.sql

-- =============================================================================
-- Model  : mart_product_performance  |  Layer: Mart
-- Sources: stg_products, stg_order_items, int_orders_enriched
-- Grain  : one row per product_id  |  Excludes: cancelled orders
-- Rankings: revenue_rank | category_revenue_rank | quality_adjusted_rank
-- =============================================================================

WITH product_sales AS (
    SELECT
        oi.product_id, p.category_name_en AS category,
        COUNT(DISTINCT oi.order_id)                           AS total_orders,
        COUNT(oi.item_sequence)                               AS units_sold,
        ROUND(SUM(oi.item_price)::numeric, 2)                 AS total_revenue,
        ROUND(AVG(oi.item_price)::numeric, 2)                 AS avg_unit_price,
        ROUND(SUM(oi.freight_cost)::numeric, 2)               AS total_freight,
        ROUND(AVG(oe.avg_review_score)::numeric, 2)           AS avg_review_score,
        COUNT(CASE WHEN oe.has_low_score THEN 1 END)          AS low_score_order_count,
        ROUND(COUNT(CASE WHEN oe.has_low_score THEN 1 END) * 100.0
            / NULLIF(COUNT(DISTINCT oi.order_id), 0), 1)      AS low_score_rate_pct,
        COUNT(CASE WHEN oe.is_late_delivery THEN 1 END)       AS late_delivery_count,
        ROUND(COUNT(CASE WHEN oe.is_late_delivery THEN 1 END) * 100.0
            / NULLIF(COUNT(DISTINCT oi.order_id), 0), 1)      AS late_delivery_rate_pct
    FROM stg_order_items oi
    JOIN stg_products p         ON p.product_id = oi.product_id
    JOIN int_orders_enriched oe ON oe.order_id  = oi.order_id
    WHERE oe.order_status != 'canceled'
    GROUP BY oi.product_id, p.category_name_en
),
category_totals AS (
    SELECT category, SUM(total_revenue) AS category_total_revenue
    FROM product_sales GROUP BY category
)
SELECT
    ps.product_id, ps.category, ps.total_orders, ps.units_sold,
    ps.total_revenue, ps.avg_unit_price, ps.total_freight,
    ps.avg_review_score, ps.low_score_order_count, ps.low_score_rate_pct,
    ps.late_delivery_count, ps.late_delivery_rate_pct,
    ROUND(ps.total_revenue * 100.0 / NULLIF(ct.category_total_revenue, 0), 2)
                                                             AS category_revenue_share_pct,
    RANK() OVER (ORDER BY ps.total_revenue DESC)             AS revenue_rank,
    RANK() OVER (PARTITION BY ps.category ORDER BY ps.total_revenue DESC)
                                                             AS category_revenue_rank,
    RANK() OVER (ORDER BY (ps.avg_review_score * ps.total_revenue) DESC NULLS LAST)
                                                             AS quality_adjusted_rank
FROM product_sales ps
JOIN category_totals ct ON ct.category = ps.category
ORDER BY revenue_rank;

5   Loading Strategy — Incremental vs Full Refresh

How raw data is loaded into each layer, and why

5.1 Strategy per Layer

LayerStrategyUnique Key / WatermarkRationale
Staging
stg_*		 Incremental — Deduped
(unique_key merge)		 order_id, customer_id, product_id, etc. New orders arrive daily; existing orders can update status (processing → delivered). Dedup-on-unique-key captures both inserts and updates without full scans.
Intermediate
int_*		 Incremental — Watermark
(filter on purchased_at)		 purchased_at >= max(purchased_at) - 3 days Derived from staging; only re-processes orders whose purchase date is within the lookback window. The 3-day buffer handles late-arriving payment or review records.
Mart
mart_*		 Full Refresh N/A (full table rebuild) Aggregations (MoM growth, customer segments, rankings) are non-idempotent with incremental loading — a new order in January would change January's total even after it was already computed. Full refresh guarantees consistency. Mart tables are small enough for this to be fast.
Exception: mart_customer_segments — Snapshot Strategy Recommended For production use, mart_customer_segments should use a Type 2 SCD snapshot pattern: store a daily snapshot of each customer's segment, not just the current state. This allows tracking of segment transitions over time (e.g., "how many customers moved from 'returning' to 'at_risk' this quarter?"). For this test, a full refresh is used for simplicity.

5.2 dbt Materialisation Config

-- dbt_project.yml (materialisation defaults per layer)
models:
  olist_dbt:
    staging:
      +materialized: incremental
      +incremental_strategy: merge
      +unique_key: order_id          # varies per model

    intermediate:
      +materialized: incremental
      +incremental_strategy: delete+insert
      +partition_by:
        field: purchased_at
        data_type: timestamp
        granularity: day

    mart:
      +materialized: table           # full refresh on every run
      +post-hook: "ANALYZE {{ this }}"  # update planner statistics after refresh

5.3 Late-Arriving Data Handling

In e-commerce pipelines, two types of late-arriving records are common:

  1. Late payments — A payment record can arrive hours after the order is created (async payment processing). The 3-day lookback buffer in the intermediate layer ensures these are captured without a full reprocessing.
  2. Late reviews — Customers can submit reviews days or weeks after delivery. For review-dependent metrics (avg_review_score, has_low_score), mart models should be considered eventually-consistent rather than point-in-time accurate.
Idempotency Guarantee All models are designed to be idempotent: running the same dbt run twice produces the same result. Incremental models use unique_key merge (not append) to prevent duplicate rows. Mart models use full refresh, which is inherently idempotent.

7   Data Quality — Checks at Every Layer

Preventive tests that catch bad data before it reaches the dashboard

7.1 Philosophy: Test Early, Fail Fast

Data quality checks are placed as early in the pipeline as possible. A broken row in staging is cheaper to catch than a broken KPI in a Senior Manager dashboard. dbt's schema.yml tests run automatically after each model build using dbt build (which runs models + tests in dependency order).

7.2 Staging Layer — Structural Validity

At this layer, checks focus on the shape of source data: are primary keys unique? Are mandatory fields populated? Do foreign keys resolve?

8   Schema Change Handling

How the pipeline adapts when the source database schema evolves

8.1 Types of Schema Changes and Their Risk

Change TypeExampleRisk LevelHandling Strategy
Add column New is_gift field added to orders Low dbt on_schema_change: sync_all_columns auto-adds new columns to the incremental model target table
Rename column order_status → status High Source freshness check + not_null test on renamed column will fail immediately, alerting the team. Fix in staging before re-running.
Drop column payment_installments removed High dbt will error on the first run that references the dropped column. CI/CD lint check (dbt compile) catches this before production.
Type change payment_value: float → text High Explicit ::numeric casts in staging will raise a cast error, making the failure visible immediately rather than silently corrupting data.
New status value New order_status = 'returned' Medium accepted_values test on stg_orders will warn/fail. Team updates the test and any relevant CASE WHEN logic in staging/intermediate.

8.2 dbt Configuration for Schema Evolution

-- In staging models: handle new columns gracefully
{{
  config(
    materialized      = 'incremental',
    unique_key        = 'order_id',
    on_schema_change  = 'sync_all_columns',  -- auto-add new source columns
    contract = {enforced: true}              -- fail if expected columns are missing
  )
}}
on_schema_change Options Trade-off sync_all_columns automatically propagates new source columns downstream — convenient but risks exposing unexpected fields in BI tools. A safer alternative is ignore (default), which requires explicit model changes for each schema update. For this Olist dataset (stable schema), sync_all_columns is appropriate. For a rapidly-evolving production source, prefer ignore with a PR review process.

8.3 Contract Testing — Enforcing Expected Schema

# schema.yml — model contracts (dbt 1.5+)
models:
  - name: stg_orders
    config:
      contract:
        enforced: true     # dbt will fail if column types don't match
    columns:
      - name: order_id
        data_type: text
      - name: purchased_at
        data_type: timestamp without time zone
      - name: order_status
        data_type: text

Model contracts act as a schema handshake between staging and intermediate layers. If a staging model changes its output shape, any intermediate model that depends on it will fail immediately at compile time — not silently at runtime.

9.5 Full DAG Code — otten_dbt_dag.py

The complete Airflow DAG with phase annotations. Stored at airflow/dags/otten_dbt_dag.py and auto-discovered by the Airflow scheduler on container startup.

# otten_dbt_dag.py — Full ELT Pipeline DAG
import os
from datetime import datetime, timedelta
from airflow import DAG
from airflow.operators.empty import EmptyOperator
from airflow.operators.bash import BashOperator
from cosmos import DbtTaskGroup, ProjectConfig, ProfileConfig, ExecutionConfig, RenderConfig, LoadMode
from cosmos.profiles import PostgresUserPasswordProfileMapping

# Path to dbt project inside the Airflow container
DEFAULT_DBT_ROOT_PATH = os.getenv("DBT_ROOT_PATH", "/opt/airflow/dags/dbt_pipeline")
DBT_EXECUTABLE_PATH  = os.getenv("DBT_EXECUTABLE_PATH", "/home/airflow/.local/bin/dbt")

# Profile: reads PostgreSQL credentials from Airflow Connection 'postgres_default'
profile_config = ProfileConfig(
    profile_name="otten_coffee_dwh",
    target_name="dev",
    profile_mapping=PostgresUserPasswordProfileMapping(
        conn_id="postgres_default",
        profile_args={"schema": "public"},
    )
)

default_args = {
    "owner": "BI_Engineer",
    "retries": 1,
    "retry_delay": timedelta(minutes=5),
}

with DAG(
    dag_id="otten_coffee_full_elt_pipeline",
    default_args=default_args,
    start_date=datetime(2024, 1, 1),
    schedule_interval="0 2 * * *",   # Daily at 02:00
    catchup=False,
    max_active_runs=1,
    tags=["otten-coffee", "elt", "dbt", "cosmos"],
) as dag:

    # ── PHASE 1: EXTRACT & LOAD ──────────────────────────────────────
    start_pipeline    = EmptyOperator(task_id="start_pipeline")

    extract_source_data = BashOperator(
        task_id="extract_postgres_data_via_airbyte",
        bash_command="echo 'Simulating Airbyte extraction...'; sleep 2"
    )

    load_raw_to_dwh = BashOperator(
        task_id="load_raw_data_to_dwh",
        bash_command="echo 'Loading raw data to public schema...'; sleep 2"
    )

    # ── PHASE 2: TRANSFORM (dbt via Astronomer Cosmos) ───────────────
    dbt_transformations = DbtTaskGroup(
        group_id="dbt_transformations",
        project_config=ProjectConfig(dbt_project_path=DEFAULT_DBT_ROOT_PATH),
        profile_config=profile_config,
        execution_config=ExecutionConfig(dbt_executable_path=DBT_EXECUTABLE_PATH),
        render_config=RenderConfig(
            load_method=LoadMode.DBT_LS,   # fast parse — no DB connection at dag-parse time
            select=["path:models"]          # run all models in the models/ directory
        )
    )

    # ── PHASE 3: QUALITY CHECK & NOTIFICATION ────────────────────────
    data_quality_check = BashOperator(
        task_id="run_data_quality_tests",
        bash_command="echo 'Running DQ constraints...'; sleep 2"
    )

    send_slack_alert = BashOperator(
        task_id="send_slack_notification",
        bash_command="echo 'Pipeline completed. Sending Slack alert...'",
        trigger_rule="all_success"  # only fires if ALL upstream tasks succeeded
    )

    end_pipeline = EmptyOperator(task_id="end_pipeline")

    # ── DEPENDENCIES ─────────────────────────────────────────────────

9.6 Astronomer Cosmos Integration

Astronomer Cosmos is the bridge between Apache Airflow and dbt. Instead of running dbt run as a single black-box BashOperator, Cosmos parses the dbt project and creates one Airflow task per dbt model — preserving the full inter-model dependency graph inside the DAG.

Why Cosmos instead of a plain BashOperator?

DimensionPlain BashOperator (dbt run)Astronomer Cosmos (DbtTaskGroup)
Visibility in Airflow UI1 task (black box)1 task per dbt model (12 tasks for this project)
Retry granularityRe-runs all models from scratchRe-runs only the failed model + downstream
Dependency graphHidden inside dbtVisible in Airflow UI as task graph
Partial successNot possibleHealthy models stay "success", failed model retried
Credential managementHardcoded profiles.yml or env varsReads from Airflow Connection — centralised secrets

Key Configuration Objects

ProjectConfig — Where is the dbt project?

ProjectConfig(dbt_project_path="/opt/airflow/dags/dbt_pipeline") — Points to the dbt project directory mounted inside the container. Cosmos reads dbt_project.yml to discover all models and their materialization configs.

ProfileConfig — How does dbt connect to PostgreSQL?

PostgresUserPasswordProfileMapping(conn_id="postgres_default", profile_args={"schema": "public"}) — Cosmos reads the Airflow Connection named postgres_default and injects credentials into dbt's connection profile at runtime. No credentials stored in code.

RenderConfig — How is the DAG parsed?

RenderConfig(load_method=LoadMode.DBT_LS, select=["path:models"]) — DBT_LS uses dbt ls to discover models (fast, no DB connection needed at parse time). select=["path:models"] runs all models under the models/ directory, in dependency order.

  otten_coffee:

Key Script Logic

# import_csv_to_postgres.py — key sections

def get_date_cols(table_name):
    """Maps tables to their date columns for proper datetime parsing."""
    date_cols_map = {
        "orders": ["order_purchase_timestamp", "order_approved_at",
                   "order_delivered_carrier_date", "order_delivered_customer_date",
                   "order_estimated_delivery_date"],
        "order_reviews": ["review_creation_date", "review_answer_timestamp"],
    }
    return date_cols_map.get(table_name, [])

def main():
    engine = create_engine("postgresql+psycopg2://postgres:<password>@localhost:5432/otten_coffee")

    for file in glob.glob("*.csv"):
        table_name = os.path.splitext(file)[0]
        df = pd.read_csv(file, parse_dates=get_date_cols(table_name))

        # DROP CASCADE removes any dbt views that reference this table
        # — prevents "cannot drop table because other objects depend on it" errors
        with engine.begin() as conn:
            conn.execute(text(f'DROP TABLE IF EXISTS "{table_name}" CASCADE'))

        df.to_sql(table_name, engine, if_exists='replace', index=False, chunksize=10000)
        # chunksize=10000 prevents memory spikes on large tables (geolocation: 1M+ rows)
Why DROP CASCADE Before Re-import? dbt creates VIEWs on top of the raw PostgreSQL tables (staging + intermediate layers are materialized as VIEW). PostgreSQL prevents dropping a table that has views depending on it. DROP TABLE IF EXISTS ... CASCADE removes the table and all dependent views in one operation, allowing a clean re-import. dbt recreates the views on the next run automatically.

    Host: host.docker.internal  |  Port: 5432  |  Database: otten_coffee  |  Schema: public
    host.docker.internal resolves to the host machine's PostgreSQL from inside the container
  8. Trigger the DAG:
    DAGs → otten_coffee_full_elt_pipeline → Toggle ON → Trigger DAG ▶
