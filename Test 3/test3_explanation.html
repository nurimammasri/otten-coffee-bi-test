1.2 Why This Three-Layer Architecture?

Each layer has a single, clearly bounded responsibility. This separation yields four concrete benefits: debuggability (isolate failures to one layer), reusability (multiple mart models can share one intermediate model), testability (data quality checks can be scoped per layer), and maintainability (changing business logic in one model does not break unrelated models).

2   Staging Layer (stg_*)

Clean mirror of source — rename, cast, null-handle. Zero business logic.

2.1 Design Rules Applied to All Staging Models

RuleImplementationRationale
Rename columns Source names → clear snake_case
(e.g., order_purchase_timestamp → purchased_at)		 Downstream models reference clean names; source typos are fixed once here
Explicit type casts price::numeric(10,2), review_score::int Prevents implicit cast errors in downstream aggregations; makes schema self-documenting
Null filtering WHERE order_id IS NOT NULL Orphaned records without PKs would corrupt downstream joins
No business logic No CASE WHEN, no joins between source tables (except stg_products) Keeps staging as a stable, predictable foundation; business logic changes belong in int_*
Empty string → NULL NULLIF(TRIM(review_comment_title), '') Standardises absence of data; avoids '' vs NULL inconsistency in analytics

2.2 Notable Design Decisions Per Model

stg_products — Exception: One JOIN Allowed

Why stg_products joins the translation table

The product_category_name_translation table is a pure reference/lookup table with no business semantics — it simply maps Portuguese names to English equivalents. Including this join in staging eliminates the need to replicate it in every downstream model that needs English category names. A LEFT JOIN is used (not INNER) to preserve products with untranslated categories, which fall back to 'uncategorized' via COALESCE.

Exception criteria: A staging JOIN is acceptable when the joined table is a static reference/lookup (no business logic, rarely changes, no PK ambiguity).

stg_products — Typo Fix in Source Column Names

-- Source has typos: "lenght" instead of "length" (missing 't')
p.product_name_lenght        AS product_name_length,
p.product_description_lenght AS description_length

The original Olist dataset has misspelled column names. Staging is the only place where these are corrected — all downstream models use the clean _length spelling.

stg_order_reviews — Derived boolean: is_low_score

(review_score <= 2) AS is_low_score

This is the only derived column in staging. It is placed here because: (1) it is a simple, non-contextual rule (score ≤ 2 = poor), (2) it will be consumed by multiple downstream models (int_orders_enriched, mart_product_performance), and (3) it requires no joins to compute. If the threshold were to change, it would only need updating in one place.

stg_order_payments — Keeps Raw Payment Grain

Design Note: One Row Per Payment Record A single order can have multiple payment rows (e.g., credit card + voucher, or instalments recorded separately). stg_order_payments preserves this grain. Aggregation to order-level (total_payment, primary_payment_type) happens in int_orders_enriched, where the business context of "per-order payment" is appropriate.

3   Intermediate Layer (int_*)

Business logic, cross-model joins, and derived metrics. The "facts" of the warehouse.

3.1 Design Philosophy

The intermediate layer is where domain knowledge lives. Models here answer the question: "What does the business need to know about each entity, combining all relevant context?" Each intermediate model has a clearly defined grain (one row per order, seller, or customer) and provides reusable, pre-computed metrics that multiple mart models can consume.

Intermediate models are never queried directly by BI tools. They are stepping stones — too granular for a Senior Manager dashboard, but too aggregated for a staging model. Their purpose is to enable mart models to be simple SELECT + GROUP BY operations.

3.2 int_orders_enriched — The Central Fact Model

Grain: One row per order_id

Sources joined: stg_orders ⋈ stg_customers ⋈ stg_order_items (aggregated) ⋈ stg_order_payments (aggregated) ⋈ stg_order_reviews (aggregated)

Key Derived Metrics

ColumnFormulaBusiness Meaning
delivery_delay_days EPOCH(delivered_at - estimated_delivery_at) / 86400 Positive = late by N days. Negative = early. NULL = not yet delivered. Rounds to 1 decimal for precision without noise.
is_late_delivery delivered_at > estimated_delivery_at Boolean flag for easy COUNT/SUM in aggregations downstream. Avoids repeating the date comparison in every mart model.
total_delivery_days EPOCH(delivered_at - purchased_at) / 86400 End-to-end fulfilment time from customer perspective. Different from delay_days which measures against the promise.
primary_payment_type Payment type with highest payment_amount in order Identifies dominant payment method for orders split across multiple payment types.
has_low_score BOOL_OR(is_low_score) from review_agg TRUE if any review for this order is 1–2 stars. Used as a quality flag in product and seller analytics.

Why Use CTEs Within the Model (not subqueries)?

WITH order_agg AS (
    SELECT order_id, SUM(item_price) AS total_items_price ...
    FROM stg_order_items GROUP BY order_id
),
payment_agg AS ( ... ),
review_agg  AS ( ... )
SELECT o.*, c.*, oa.*, pa.*, ra.*, -- delivery metrics ...
FROM stg_orders o
JOIN stg_customers c ON ...
LEFT JOIN order_agg oa ON oa.order_id = o.order_id ...

Each upstream model (order_items, payments, reviews) is aggregated to order-grain in its own CTE before joining to orders. This prevents the fan-out problem — if orders were joined to non-aggregated items first, each order would appear N times (once per item), causing inflated aggregations downstream.

3.3 int_seller_metrics — Operational Performance per Seller

Grain: One row per seller_id

Aggregates all completed orders containing items from each seller. Provides: revenue, volume, customer satisfaction (avg review score, low-score rate), and delivery performance (late rate, avg days late). Also identifies each seller's top category by revenue using DISTINCT ON.

3.4 int_customer_metrics — Lifetime Behaviour per Customer

Grain: One row per customer_unique_id

Derives RFM-adjacent metrics: total orders, total spend, first/last order dates, customer lifetime days, avg days between purchases (frequency), and preferred state. These feed directly into mart_customer_segments.

avg_days_between_orders uses (lifetime_days) / (total_orders - 1) to measure true inter-purchase intervals, and returns NULL for one-time buyers (where the denominator would be 0).

4   Mart Layer (mart_*)

Business-ready aggregations — directly consumed by BI dashboards and Senior Manager reports

4.1 Design Philosophy

Mart models are designed for BI consumption: wide, flat, pre-aggregated tables that a Senior Manager or a BI tool can query without complex joins. Each mart model answers a specific business question, and column names are chosen for clarity over technical precision.

4.3 mart_customer_segments

Business Question: "Who are our customers and how do we retain them?"

Classifies every customer into one of four segments based on purchase recency and frequency. Designed to drive CRM targeting and retention campaign prioritisation.

SegmentRuleCount (dataset)Avg Spend (R$)
newExactly 1 order ever92,635161.56
returning2+ orders, last order ≤ 180 days ago1,239321.31
at_risk2+ orders, last order 181–365 days ago1,136303.39
churned2+ orders, last order > 365 days ago549285.31
Critical Business Insight from Segmentation 96.8% of customers (92,635 out of ~95,500) made only one purchase — they are classified as "new" indefinitely. Returning customers spend 2× more (R$ 321 vs R$ 162) than one-time buyers. This makes retention the highest-ROI business initiative: converting even 1% of new customers to returning adds ~R$ 15M in incremental revenue.

Why use dataset max date (not NOW()) as reference?
The segmentation uses MAX(purchased_at) from the dataset as the reference date instead of CURRENT_TIMESTAMP. This makes the model reproducible: running the same query twice at different times produces the same result for historical analysis. Using NOW() would continuously re-classify customers as time passes, making historical snapshots inconsistent.

4.4 mart_product_performance

Business Question: "Which products drive revenue and which need attention?"

Provides product-level metrics combining commercial performance (revenue, units sold) with quality signals (review scores, low-score rates, late delivery rates). Includes three ranking dimensions:

  • revenue_rank — global rank across all products
  • category_revenue_rank — rank within the product's category (enables "best in category" analysis)
  • quality_adjusted_rank — ranks by avg_review_score × total_revenue, penalising high-revenue products with poor ratings

The quality-adjusted ranking is designed to surface products that are commercially important but risk eroding brand trust — a signal for Senior Managers to investigate product quality issues.

5   Loading Strategy — Incremental vs Full Refresh

How raw data is loaded into each layer, and why

5.1 Strategy per Layer

LayerStrategyUnique Key / WatermarkRationale
Staging
stg_*		 Incremental — Deduped
(unique_key merge)		 order_id, customer_id, product_id, etc. New orders arrive daily; existing orders can update status (processing → delivered). Dedup-on-unique-key captures both inserts and updates without full scans.
Intermediate
int_*		 Incremental — Watermark
(filter on purchased_at)		 purchased_at >= max(purchased_at) - 3 days Derived from staging; only re-processes orders whose purchase date is within the lookback window. The 3-day buffer handles late-arriving payment or review records.
Mart
mart_*		 Full Refresh N/A (full table rebuild) Aggregations (MoM growth, customer segments, rankings) are non-idempotent with incremental loading — a new order in January would change January's total even after it was already computed. Full refresh guarantees consistency. Mart tables are small enough for this to be fast.
Exception: mart_customer_segments — Snapshot Strategy Recommended For production use, mart_customer_segments should use a Type 2 SCD snapshot pattern: store a daily snapshot of each customer's segment, not just the current state. This allows tracking of segment transitions over time (e.g., "how many customers moved from 'returning' to 'at_risk' this quarter?"). For this test, a full refresh is used for simplicity.

5.2 dbt Materialisation Config

-- dbt_project.yml (materialisation defaults per layer)
models:
  olist_dbt:
    staging:
      +materialized: incremental
      +incremental_strategy: merge
      +unique_key: order_id          # varies per model

    intermediate:
      +materialized: incremental
      +incremental_strategy: delete+insert
      +partition_by:
        field: purchased_at
        data_type: timestamp
        granularity: day

    mart:
      +materialized: table           # full refresh on every run
      +post-hook: "ANALYZE {{ this }}"  # update planner statistics after refresh

5.3 Late-Arriving Data Handling

In e-commerce pipelines, two types of late-arriving records are common:

  1. Late payments — A payment record can arrive hours after the order is created (async payment processing). The 3-day lookback buffer in the intermediate layer ensures these are captured without a full reprocessing.
  2. Late reviews — Customers can submit reviews days or weeks after delivery. For review-dependent metrics (avg_review_score, has_low_score), mart models should be considered eventually-consistent rather than point-in-time accurate.
Idempotency Guarantee All models are designed to be idempotent: running the same dbt run twice produces the same result. Incremental models use unique_key merge (not append) to prevent duplicate rows. Mart models use full refresh, which is inherently idempotent.

6   Orchestration Tool — Choice & Justification

Scheduling, monitoring, and dependency management for the ELT pipeline

6.1 Tool Comparison

dbt Cloud RECOMMENDED
  • Native dbt integration — zero configuration overhead
  • Built-in scheduler (cron expressions)
  • Web UI for run monitoring, logs, and alerts
  • Automatic lineage graph and documentation generation
  • Slim CI/CD: only runs changed models + downstream
  • dbt Cloud IDE for collaborative model development
  • Managed infrastructure — no DevOps required
Apache Airflow ALTERNATIVE
  • More powerful for multi-system orchestration
  • Can trigger extracts (e.g., Airbyte, Fivetran) before dbt runs
  • Custom retry logic, branching, and complex DAG patterns
  • Open source — no licensing cost
  • Requires infrastructure setup and maintenance (Kubernetes / Compose)
  • Use astronomer-cosmos for native dbt task integration
  • Better for multi-source pipelines (not just PostgreSQL → PostgreSQL)

6.2 Why dbt Cloud is Recommended for This Use Case

7   Data Quality — Checks at Every Layer

Preventive tests that catch bad data before it reaches the dashboard

7.1 Philosophy: Test Early, Fail Fast

Data quality checks are placed as early in the pipeline as possible. A broken row in staging is cheaper to catch than a broken KPI in a Senior Manager dashboard. dbt's schema.yml tests run automatically after each model build using dbt build (which runs models + tests in dependency order).

7.2 Staging Layer — Structural Validity

At this layer, checks focus on the shape of source data: are primary keys unique? Are mandatory fields populated? Do foreign keys resolve? 

# schema.yml — staging layer tests

models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: customer_id
        tests:
          - not_null
          - relationships:
              to: ref('stg_customers')
              field: customer_id
      - name: order_status
        tests:
          - accepted_values:
              values: ['delivered','shipped','canceled','invoiced',
                       'processing','created','approved','unavailable']
      - name: purchased_at
        tests:
          - not_null
          - dbt_utils.expression_is_true:
              expression: "purchased_at >= '2016-01-01'"

  - name: stg_order_reviews
    columns:
      - name: score
        tests:
          - not_null
          - accepted_values:
              values: [1, 2, 3, 4, 5]

  - name: stg_order_payments
    columns:
      - name: payment_amount
        tests:
          - dbt_utils.expression_is_true:
              expression: "payment_amount >= 0"

7.3 Intermediate Layer — Business Logic Validity

At this layer, checks validate derived metrics and joined data integrity. The key question: "Do computed fields make business sense?" 

# schema.yml — intermediate layer tests

  - name: int_orders_enriched
    tests:
      - dbt_utils.unique_combination_of_columns:
          combination_of_columns: [order_id]   # one row per order
    columns:
      - name: total_payment
        tests:
          - dbt_utils.expression_is_true:
              expression: "total_payment >= 0"
      - name: item_count
        tests:
          - dbt_utils.expression_is_true:
              expression: "item_count > 0"
      - name: delivery_delay_days
        tests:
          - dbt_utils.expression_is_true:
              expression: "delivery_delay_days BETWEEN -365 AND 365 OR delivery_delay_days IS NULL"

  - name: int_customer_metrics
    tests:
      - dbt_utils.unique_combination_of_columns:
          combination_of_columns: [customer_unique_id]
    columns:
      - name: total_spend
        tests:
          - dbt_utils.expression_is_true:
              expression: "total_spend > 0"

7.4 Mart Layer — Completeness & Anomaly Detection

At the mart layer, checks ensure the output is complete, non-empty, and within expected ranges. These are the last line of defence before BI tools consume the data.

7.5 Source Freshness Check

In addition to row-level tests, dbt's source freshness checks verify that the source database is being updated as expected — catching pipeline failures upstream of dbt. 

# sources.yml
sources:
  - name: olist
    database: postgres
    schema: public
    loaded_at_field: order_purchase_timestamp
    freshness:
      warn_after:  {count: 24,  period: hour}
      error_after: {count: 48,  period: hour}
    tables:
      - name: orders
      - name: order_payments
      - name: order_reviews

8   Schema Change Handling

How the pipeline adapts when the source database schema evolves

8.1 Types of Schema Changes and Their Risk

Change TypeExampleRisk LevelHandling Strategy
Add column New is_gift field added to orders Low dbt on_schema_change: sync_all_columns auto-adds new columns to the incremental model target table
Rename column order_status → status High Source freshness check + not_null test on renamed column will fail immediately, alerting the team. Fix in staging before re-running.
Drop column payment_installments removed High dbt will error on the first run that references the dropped column. CI/CD lint check (dbt compile) catches this before production.
Type change payment_value: float → text High Explicit ::numeric casts in staging will raise a cast error, making the failure visible immediately rather than silently corrupting data.
New status value New order_status = 'returned' Medium accepted_values test on stg_orders will warn/fail. Team updates the test and any relevant CASE WHEN logic in staging/intermediate.

8.2 dbt Configuration for Schema Evolution

-- In staging models: handle new columns gracefully
{{
  config(
    materialized      = 'incremental',
    unique_key        = 'order_id',
    on_schema_change  = 'sync_all_columns',  -- auto-add new source columns
    contract = {enforced: true}              -- fail if expected columns are missing
  )
}}
on_schema_change Options Trade-off sync_all_columns automatically propagates new source columns downstream — convenient but risks exposing unexpected fields in BI tools. A safer alternative is ignore (default), which requires explicit model changes for each schema update. For this Olist dataset (stable schema), sync_all_columns is appropriate. For a rapidly-evolving production source, prefer ignore with a PR review process.

8.3 Contract Testing — Enforcing Expected Schema

# schema.yml — model contracts (dbt 1.5+)
models:
  - name: stg_orders
    config:
      contract:
        enforced: true     # dbt will fail if column types don't match
    columns:
      - name: order_id
        data_type: text
      - name: purchased_at
        data_type: timestamp without time zone
      - name: order_status
        data_type: text

Model contracts act as a schema handshake between staging and intermediate layers. If a staging model changes its output shape, any intermediate model that depends on it will fail immediately at compile time — not silently at runtime.

  • Three-layer separation of concerns with explicit contracts between layers
