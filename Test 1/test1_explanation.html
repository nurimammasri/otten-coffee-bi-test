1   Dataset & Global Assumptions

Brazilian E-Commerce (Olist) — Public PostgreSQL on Supabase

1.1 Dataset Overview

The dataset is the publicly available Olist Brazilian E-Commerce dataset, hosted on a read-only PostgreSQL instance on Supabase. It covers approximately two years of e-commerce transactions (September 2016 – August 2018) across Brazil.

1.2 Global Assumptions Applied Across All Queries

Revenue Source

Revenue is always taken from order_payments.payment_value, not order_items.price. Payment value captures the actual amount charged to the customer, including any applied discounts, vouchers, or installment adjustments. This gives a more accurate picture of real business revenue.

Order Status Filtering

All analyses exclude orders with order_status = 'canceled'. Cancelled orders represent intent without transaction — including them would inflate order counts and distort revenue metrics. Other statuses (delivered, shipped, processing, invoiced, approved, created) are included unless otherwise noted.

Customer Identity

The customers table has two identifier columns. customer_id is unique per order (a returning customer can have multiple customer_id values). For customer-level analysis, customer_unique_id is used — this is the true cross-order customer identifier. This is critical for cohort retention analysis.

Data Gaps

November 2016 has zero orders in the dataset (only Sep, Oct, Dec 2016 have data). September 2018 has only a handful of records, suggesting data collection ended mid-month. These gaps are reflected as anomalous MoM changes and do not indicate real business events.

2   A1 — Monthly Revenue Trend

Total revenue per month with month-over-month percentage change

2.1 Query Logic

The query uses a two-step CTE pattern. The inner CTE monthly_revenue groups payments by purchase year and month, then the outer SELECT applies a LAG() window function to access the previous month's revenue without a self-join — keeping the query efficient and readable.

Why LAG() over a self-join?

A self-join approach would require joining the aggregated monthly table to itself, which means two full scans of the aggregated result. LAG() accesses the prior row in a single ordered window pass — cleaner SQL and more efficient execution.

MoM Change Formula

mom_change_pct = ROUND(
    (current_month - previous_month) * 100.0
    / NULLIF(previous_month, 0)
, 2)

NULLIF(prev, 0) prevents division-by-zero for any month with zero revenue. The result is NULL for the first month (no prior period to compare).

2.3 Key Business Insights

Insight 1 — November 2017: Peak Revenue Month November 2017 recorded the highest single-month revenue at R$ 1,187,224 — a +53.6% jump from October. This almost certainly coincides with Black Friday Brazil (last Friday of November), a major retail event. The dip in December (+26% below November) is consistent with post-peak normalisation seen in e-commerce.
Insight 2 — Consistent Growth Phase (Jan–Nov 2017) Revenue grew from R$ 138K (Jan 2017) to R$ 1.19M (Nov 2017) — an 8.6× increase in 11 months. This reflects Olist's rapid marketplace expansion during this period. Only April and June 2017 showed slight month-over-month dips.
Insight 3 — Plateau in 2018 After January 2018, revenue stabilised around R$ 1–1.15M/month with very small MoM changes (±1%). This plateau suggests the business reached a steady state or that growth was tapering. Without YoY data beyond 2018, it is unclear whether this is seasonal or structural.
Data Artefact — Dec 2016 and Sep 2018 December 2016 shows R$ 19.62 (1 transaction) and September 2018 shows R$ 166.46. These are data collection boundaries, not real business drops. MoM changes for adjacent months should be interpreted with caution.

3   A2 — Top 10 Product Categories by Revenue

Ranked by total payment value, excluding cancelled orders

3.1 Query Logic

The query chains five tables: products → product_category_name_translation → order_items → orders → order_payments. An INNER JOIN to the translation table is intentional — it excludes products with no English category mapping (null or untranslatable), ensuring the output is fully human-readable.

avg_order_value Calculation

avg_order_value = total_revenue / COUNT(DISTINCT order_id) — this represents the average revenue per order that involved at least one item from this category. It is not a per-item price; an order can contain items from multiple categories.

Why use payment_value instead of item price?

order_payments.payment_value is used for revenue because it reflects the actual cash received, including voucher redemptions and installment adjustments. Using order_items.price would overstate revenue for orders where vouchers were applied.

4   A3 — Customer Cohort Retention

How many customers from each acquisition month returned in M+1, M+2, M+3

4.1 Query Logic

The analysis uses a 3-CTE pipeline. Each step has a clear, single responsibility:

CTEPurpose
customer_orders Deduplicate to one row per (customer_unique_id, order_month). Uses DISTINCT to prevent double-counting customers who placed multiple orders in the same month.
cohorts Identifies each customer's acquisition cohort as their earliest order month using MIN().
month_diff Calculates the integer month offset between each subsequent order and the cohort month.

Why use YEAR×12 + MONTH arithmetic instead of interval comparison?

months_after_cohort =
    (YEAR(order_month) * 12 + MONTH(order_month))
  - (YEAR(cohort_month) * 12 + MONTH(cohort_month))

Interval-based approaches (e.g., subtracting timestamps and dividing by 30 days) produce floating-point values that can mis-classify a purchase made on Feb 28 vs Mar 1. Integer year-month arithmetic guarantees exact month offsets regardless of month length.

LEFT JOIN for retention counting

A LEFT JOIN from cohorts to month_diff ensures that cohorts with zero retained customers still appear in the output (retained_m1/m2/m3 = 0 via COUNT of NULL). An INNER JOIN would silently drop those cohorts.

4.3 Key Business Insights

Critical Insight — Extremely Low Repeat Purchase Rate Across all cohorts, the M+1 retention rate consistently falls below 1%. For example, the August 2017 cohort of 4,162 customers had only 28 returnees in September — a 0.67% retention rate. This means over 99% of customers are one-time buyers.
Insight 1 — This is Normal for Marketplace E-Commerce Low retention in general e-commerce marketplaces is common: customers buy when they have a specific need, not out of habit. However, the near-zero rates here suggest an opportunity: even a 2% retention improvement doubles the retained customer base. Targeted re-engagement campaigns (email, push, personalised offers) could have significant revenue impact.
Insight 2 — No Clear Cohort Improvement Over Time Earlier cohorts (Jan–Mar 2017) show similar or lower retention than later ones, suggesting no systematic improvement in retention over the observation period. A mature marketplace would typically show improving retention as the product and recommendation engines mature.
Recommendation for Further Analysis A more complete retention picture would include: (1) percentage retention rates (not just absolute counts), (2) revenue retained per cohort, and (3) comparison of retained vs churned customer lifetime value. These would enable ROI calculation for retention campaigns.

5   Part B — Query Optimisation

Refactoring a slow multi-join category analytics query for production performance

5.1 Bottlenecks Identified

The original query was analysed structurally (without EXPLAIN ANALYZE access on this read-only instance) to identify the following bottlenecks:

1
Triple Table Scan — products + order_items
The original query defines three independent subqueries (b, ps, ls), each performing its own JOIN products → order_items from scratch. This means the database reads both large tables three separate times. On a 100K+ row order_items table, this triples the I/O cost of the query.
2
Repeated Full-Table Window Function — SUM() OVER ()
SUM(total_revenue) OVER () (grand total) appears twice in the outer SELECT: once for revenue_share_pct and once as the denominator for cumulative_revenue_pct. Each occurrence triggers a separate window pass over all rows, computing the same value redundantly.
3
Deeply Nested Subquery Structure
Three levels of nesting (innermost → b → ranked → outer) limit the query planner's ability to push down filters and choose optimal join order. Deep nesting also makes the query harder to read, debug, and maintain.
4
RANK() Computed Inside Subquery, Then More Window Functions on Top
RANK() OVER (ORDER BY SUM(price) DESC) is computed inside the subquery ranked, requiring a sort. The outer query then applies additional window functions (RANK() OVER (ORDER BY late_rate_pct DESC), cumulative sum), requiring additional sort passes over the already-sorted data.
5
COUNT(DISTINCT order_id) Recomputed for late_rate_pct
The expression ROUND(ls.late_count * 100.0 / NULLIF(COUNT(DISTINCT b.order_id), 0), 1) recalculates COUNT(DISTINCT order_id) even though this value was already computed as total_orders in the same GROUP BY. The database must re-evaluate the aggregate for the denominator.

5.2 Optimisation Strategy Applied

1
Single-Pass CTE with FILTER Aggregation
All three subqueries (b, ps, ls) are consolidated into one CTE (category_stats) with a single join chain. The payment type breakdown (credit_card, boleto) is handled using PostgreSQL's FILTER (WHERE payment_type = '...') clause — this computes multiple conditional aggregates in one pass, eliminating the separate ps subquery entirely.
2
Pre-computed Grand Total via Scalar CTE + CROSS JOIN
A separate CTE (total) pre-computes the grand total revenue once. It is joined to the main result using CROSS JOIN (one row × N rows = N rows, no expansion). Both revenue_share_pct and cumulative_revenue_pct reference t.grand_total — a single pre-computed value — instead of triggering two separate window scans.
3
Flattened Nesting — CTE-first Architecture
All intermediate computations are expressed as named CTEs. The final SELECT contains only window functions and column references — no nested subqueries. This gives the query planner maximum visibility into the full execution graph and allows it to choose optimal join strategies.
4
All Window Functions in Final SELECT Only
RANK() and the cumulative sum are moved to the outermost SELECT only, applied once to the final aggregated result set. This reduces the number of sort operations from multiple passes to a single ordered window evaluation over the compact grouped result.
5
Reuse Computed Columns
late_rate_pct is computed by referencing cs.total_orders (already computed in the CTE) rather than re-evaluating COUNT(DISTINCT order_id). Similarly, avg_order_value reuses cs.total_revenue / cs.total_orders.

5.3 Index Recommendations

These indexes would significantly accelerate the optimised query in a writable environment: 

-- Enable fast product → item joins (most critical: order_items is the largest table)
CREATE INDEX idx_order_items_product_id
    ON order_items (product_id);

-- Enable fast payment type filtering without full scan
CREATE INDEX idx_order_payments_order_id_type
    ON order_payments (order_id, payment_type);

-- Enable fast delivery date comparison for late delivery calculation
CREATE INDEX idx_orders_delivery_dates
    ON orders (order_delivered_customer_date, order_estimated_delivery_date)
    WHERE order_delivered_customer_date IS NOT NULL;

-- Enable fast product category grouping
CREATE INDEX idx_products_category
    ON products (product_category_name);
Note on Partial Index The WHERE order_delivered_customer_date IS NOT NULL partial index on orders reduces index size significantly, since NULL-delivered orders are irrelevant to the delivery analysis. PostgreSQL will use this index only when the query also filters on this condition.

5.4 How to Verify the Improvement in a Real Environment

Step 1 — Capture baseline plan

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) ;

Record: cost=, actual time=, Buffers: shared hit/read, and whether scans are Seq Scan or Index Scan.

Step 2 — Run optimised query and compare

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) ;

Look for: reduction in Seq Scan nodes, lower cost= estimates, lower actual time=, fewer shared read buffer accesses.

Step 3 — Warm cache and time both queries

SELECT pg_stat_reset();  -- reset stats

-- Run each query 3 times to warm cache, then compare \timing output in psql
\timing
;
;

Step 4 — Monitor pg_stat_statements

SELECT query, mean_exec_time, total_exec_time, calls
FROM   pg_stat_statements
WHERE  query ILIKE '%category_stats%'
ORDER BY mean_exec_time DESC;

This shows real average execution time across all calls, accounting for cache warm-up.
